## 三、项目面试题（核心重点）

### 3.1 i18n 国际化管理平台（重点项目）

**Q19: 为什么要做这个 i18n 平台？解决了什么痛点？**

- 回答要点：
  - 多项目、多语言文案分散管理的问题
  - Excel 管理混乱，协作效率低
  - 手动 Google 翻译效率低下
  - 文案版本管理困难

**Q20: 技术架构是如何设计的？为什么选择这套技术栈？**

- 回答要点：
  - 前端：React（组件化、生态丰富）
  - 后端：NestJS（TypeScript、依赖注入、模块化）
  - 数据库：PostgreSQL（关系型数据、事务支持）
  - API 层：Fastify（高性能）
  - 部署：GitHub Actions + PM2（自动化部署、进程管理）

**Q21: AI 翻译功能是如何实现的？为什么选择 Qwen-plus 模型？**

- 回答要点：
  - 接入阿里云通义千问 Qwen-plus 模型
  - API 调用封装、错误处理、重试机制
  - 批量翻译优化、翻译质量控制
  - 成本考虑（相比 GPT-4 等国外模型）
  - 效率提升：从手动翻译到 AI 翻译，效率提升约 10 倍

**Q22: 多项目接入是如何设计的？如何保证数据隔离？**

- 回答要点：
  - 项目维度的数据隔离设计
  - 权限控制体系（项目管理员、成员角色）
  - JSON 导入导出功能设计
  - 版本管理机制

**Q23: 自动化部署流程是如何实现的？**

- 回答要点：
  - GitHub Actions 工作流配置
  - 前后端分离部署策略
  - PM2 进程管理（守护进程、自动重启、日志管理）
  - 环境变量管理、数据库迁移

**Q24: 这个项目遇到的最大挑战是什么？如何解决的？**

- 回答要点：
  - 可以讲并发翻译的性能优化
  - 或者讲数据一致性保证
  - 或者讲权限控制的复杂场景处理

### 3.2 PWA SDK 与基础设施（技术亮点）

**Q25: 为什么要开发 PWA SDK？解决了什么问题？**

- 回答要点：
  - 业务接入 PWA 能力成本高
  - Service Worker 生命周期管理复杂
  - Manifest 配置繁琐
  - 缓存策略需要统一规范

**Q26: PWA SDK 的核心功能有哪些？如何设计的？**

- 回答要点：
  - Service Worker 生命周期封装（注册、安装、激活、更新）
  - Manifest 配置管理
  - 静态资源缓存能力（基于 Workbox）
  - 推送通知能力
  - 应用安装引导
  - 提供统一的 API 接口，降低业务接入成本

**Q27: 缓存策略是如何设计的？为什么选择 Stale-While-Revalidate？**

- 回答要点：
  - Stale-While-Revalidate 策略：先返回缓存，后台更新
  - 适用场景：图片等静态资源，允许短暂的过期内容
  - 过期控制：maxAgeSeconds 设置
  - 容量控制：maxEntries 限制缓存数量
  - 弱网或离线场景下的用户体验提升

**Q28: 如何搭建 Verdaccio 私有 npm 仓库？如何管理版本？**

- 回答要点：
  - Verdaccio 安装与配置
  - 权限管理（发布权限、下载权限）
  - 版本管理策略（语义化版本）
  - 如何实现跨项目复用
  - 与 CI/CD 的集成

### 3.3 性能优化项目（数据亮点）

**Q29: H5 页面性能优化是如何做的？具体优化了哪些方面？**

- 回答要点（这是你的核心亮点，要详细展开）：
  - **分析阶段**：使用 webpack-bundle-analyzer 分析依赖体积
  - **优化策略**：
    1. externals + CDN 引入（vue、vue-router、SDK）
    2. 代码分割与按需加载
    3. Tree Shaking 优化
    4. 图片 WebP 转换
    5. 资源压缩（gzip/brotli）
  - **优化成果**：
    - 包体积：81.78KB → 53.06KB（↓35%）
    - FCP：2.86s → 1.59s（↓44%）
    - LCP：3.15s → 1.79s（↓43%）

**Q30: 如何衡量性能优化的效果？使用了哪些工具？**

- 回答要点：
  - Chrome DevTools（Performance、Lighthouse）
  - webpack-bundle-analyzer（包体积分析）
  - Web Vitals 指标（FCP、LCP、CLS、FID）
  - 真实用户监控（RUM）数据

### 3.4 Hybrid 架构（JSBridge）

**Q31: JSBridge 的通信机制是如何设计的？**

- 回答要点：
  - H5 与原生的双向通信设计
  - 协议定义（方法名、参数、回调）
  - 统一 Bridge 调用层封装
  - 错误处理与超时机制
  - 版本兼容性处理

**Q32: JSBridge 支持哪些核心能力？如何降低业务接入复杂度？**

- 回答要点：
  - 登录态获取
  - 埋点上报
  - 原生组件调用（相机、相册、分享等）
  - UI 控制（标题栏、导航栏）
  - 统一封装，提供简洁的 API 接口

### 3.5 导航工具平台

**Q33: 为什么选择 Next.js + Supabase 这套技术栈？**

- 回答要点：
  - Next.js：SSR/SSG、文件路由、API Routes
  - Supabase：开箱即用的数据库 + 鉴权 + 实时订阅
  - 快速开发、低运维成本
  - Vercel 部署，持续集成

**Q34: URL 参数解析工具是如何实现的？**

- 回答要点：
  - query 参数解析（URLSearchParams）
  - path 参数拆解
  - 可视化展示设计
  - 提升开发调试效率的实际场景

### 3.6 Vue 项目脚手架（shedcli）

**Q35: 为什么要开发这个脚手架？如何实现的？**

- 回答要点：
  - 解决项目初始化重复劳动
  - 支持 Vue2 和 Vue3 双版本
  - 插件化功能选择（API 域名、路由、状态管理、国际化）
